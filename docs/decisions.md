## 决策 1: 项目隔离架构设计

### 背景
初期所有代码和数据都针对华住安住REIT单项目，随着新增华润成都万象城项目，需要支持多项目并行且不混淆数据。

### 选项分析

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 代码分支 | 简单直接 | 重复代码，合并困难 |
| B: 目录隔离 | 数据物理分离 | 硬编码路径多，切换不便 |
| **C: 配置中心+路径管理** | 单一代码库，动态路径解析 | 需要设计配置层级 |

### 最终选择
**选项 C**: 创建 `run_config.yaml` 配置中心 + `ProjectConfigManager` 路径管理器

```yaml
projects:
  huazhu:
    asset_types: [hotel]
    data_dir: data/huazhu
    output_dir: output/huazhu
  huarun_chengdu:
    asset_types: [mall, hotel]
    data_dir: data/huarun_chengdu
    output_dir: output/huarun_chengdu

active_project: huazhu
```

配置优先级：命令行参数 > 环境变量 > 配置文件 > 默认值

### 违反后果
- 数据混淆：不同项目数据相互覆盖
- 输出污染：DCF结果写入错误目录
- 难以追溯：无法确定某次运行的数据版本

## 决策 2: 多业态DCF模型架构

### 背景
华润成都万象城包含商业(mall)和酒店(hotel)两种业态，需要统一计算综合体估值。

### 选项分析

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 分别运行后手工合并 | 简单 | 无法自动化，易出错 |
| B: 单一模型处理所有业态 | 无需协调 | 代码复杂度高，难以维护 |
| **C: MultiAssetDCFModel + 子模型** | 分层清晰，易于扩展 | 需要设计统一接口 |

### 最终选择
**选项 C**: `MultiAssetDCFModel` 作为协调器，`MallDCFModel`/`HotelDCFModel`作为子模型

```python
ASSET_TYPE_REGISTRY = {
    "mall": MallDCFModel,
    "hotel": HotelDCFModel,
    # 未来扩展: "industrial": IndustrialDCFModel
}

MultiAssetDCFModel:
  - 检测业态 → _detect_asset_types()
  - 构建子模型 → _build_sub_models()
  - 分别计算 → model.calculate()
  - 合并结果 → _merge_results()
```

统一接口 `BaseDCF`:
- `calculate() -> DCFResult`
- `adjust(discount_rate, growth_rate, noi_multiplier) -> BaseDCF`

### 违反后果
- 业态耦合：修改一个业态影响其他业态
- 扩展困难：新增业态需修改核心代码
- 结果不一致：子模型输出格式不统一导致合并失败

## 决策 3: NOI推导值优先策略

### 背景
模型推导的NOI与招募说明书披露的NOI/CF存在差异（广州-3.2%，上海-8.6%）。

### 选项分析

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 差异>5%时回退使用招募值 | 与招募估值对齐 | 掩盖真实成本结构差异，失去模型意义 |
| **B: 始终使用推导值，标注差异** | 反映REITs后真实成本，差异可见 | 与招募值偏差需额外解释 |

### 最终选择
**选项 B**: 始终使用推导NOI作为DCF输入

```python
# 标注策略
差异 ≤ 5%: 标注 [PASS]
差异 > 5%: 标注 [差异 X.X%]，仍使用推导值
```

理由：
1. 推导NOI反映REITs后的实际成本结构（成本重分类）
2. 与招募利润表口径的差异是口径差异，不是推导错误
3. 管理费用、物业费、保险等科目在REITs后确实会发生变化

### 违反后果
- 估值失真：使用已不适用的历史成本结构
- 缺乏可比性：不同项目使用不同口径，无法横向对比
- 审计失败：无法解释为何某些科目与招募值不一致

## 决策 4: 增值税处理方式

### 背景
初期模型从收入中单独扣除增值税，导致NOI偏低约10%。

### 选项分析

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 维持原逻辑（收入-VAT=净收入） | 符合"增值税是税"的直觉 | 与中国GAAP口径冲突，重复扣除 |
| **B: 收入已不含税，不单独扣VAT** | 符合中国会计准则，避免重复扣除 | 需要理解价税分离机制 |

### 最终选择
**选项 B**: 增值税禁止重复扣除

**酒店REIT**:
```
中国GAAP营业收入 = 不含增值税净额
税金及附加 ≠ 增值税，仅含附加税
NOI推导中禁止单独扣除增值税
```

**商业REIT**:
```
收入口径 = 不含税净额（固定租金/物管费/停车等均为不含税）
增值税 = 纯过路，不进NOI计算
附加税 = VAT × 12% → 真实税负，计入tax
```

### 违反后果
- NOI偏低：重复扣除导致估值偏低10%+
- 审计质疑：无法解释为何利润表"营业收入"还要扣增值税
- 跨项目不可比：不同项目VAT处理方式不一致

## 决策 5: 成本口径选择

### 背景
历史利润表"营业成本"与REITs明细成本存在差异。

### 选项分析

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 使用历史利润表"运营成本(不含折旧)" | 数据完整，有3年历史 | 包含公司层面成本，REITs后结构变化 |
| **B: 使用REITs明细成本** | 反映发行后成本结构，项目独立性 | 部分数据为估算值 |

### 最终选择
**选项 B**: REITs明细成本 = 运营明细 + 物业(独立) + 保险(独立)

成本口径二选一，**禁止混用**：
- 口径A: 历史利润表"运营成本" = 已含物业/保险
- 口径B: REITs明细 = 运营明细 + 物业(独立) + 保险(独立)

REITs建模使用**口径B**，理由：
1. 物业费、保险在REITs后变为独立合同，不再并入营业成本
2. 利润表"营业成本"是综合科目，REITs后结构发生变化
3. 明细成本更透明，便于审计和敏感性分析

### 违反后果
- 双重计算：同时用历史成本基数又加独立物业/保险
- 广州影响：约481万/年成本重复
- 上海影响：约162万/年成本重复

## 决策 6: 管理费计算方式

### 背景
利润表"管理费用"包含公司层面行政开支，不适用于REITs项目。

### 选项分析

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 使用利润表"管理费用" | 数据直接可得 | 包含公司行政，REITs后由基金承担 |
| **B: GOP × 管理费率(3%)** | 仅含酒店管理公司费用 | 需要明确定义管理费率 |

### 最终选择
**选项 B**: 管理费 = GOP × 3%

依据：招募说明书明确定义"付给酒店管理公司的费用，按GOP×基准费率缴纳"

对比验证：
- 广州: 管理费=397万(4.6%GOP), 历史管理费用=449万。差额52万为公司行政
- 上海: 管理费=14万(0.8%GOP), 历史管理费用=160万。差额146万为公司行政

### 违反后果
- 管理费虚高：上海偏差146万，广州偏差52万
- NOI偏低：偏差直接影响估值
- 结构错误：利润表"管理费用"口径不适用于REITs

## 决策 7: 折现约定与终值处理

### 背景
DCF模型需要明确折现时点约定和终值处理方式。

### 选项分析

**折现时点**:
| 选项 | 说明 | 影响 |
|------|------|------|
| A: 期中折现 | 假设现金流在年中发生 | 估值偏高约3% |
| **B: 期末折现** | 假设现金流在年末发生 | 与招募说明书一致 |

**终值处理**:
| 选项 | 说明 | 适用性 |
|------|------|--------|
| A: 永续增长模型 | Gordon Growth Model | 需要无限期土地使用权 |
| **B: 持有到期，无残值** | 土地使用权到期后无残值 | 符合中国土地制度 |

### 最终选择
**折现**: 期末折现，与招募说明书保持一致
**终值**: 无残值，DCF年限 = 土地使用权剩余年限（含部分年）

```
终值 = 0（土地使用权到期后无残值）
不使用永续增长模型
DCF年限 = 土地使用权剩余年限（含部分年）
折现因子 = (1+r)^(full_years + partial_fraction)
```

### 违反后果
- 估值虚高：期中折现+永续增长模型导致估值偏高15%+
- 与招募不可比：不同折现约定导致差异
- 违反土地制度：中国土地使用权到期后需重新出让

## 决策 8: 输出不可覆盖原则

### 背景
需要保留历史运行记录以便追溯和对比。

### 选项分析

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 输出到固定文件 | 简单，直接查看最新结果 | 历史结果丢失，无法对比 |
| **B: 时间戳目录 + latest软链接** | 保留历史，同时快速访问最新 | 目录结构复杂 |

### 最终选择
**选项 B**: 输出到时间戳目录，同步至latest

```
output/{project}/
├── run_YYYYMMDD_HHMMSS/    # 本次运行结果
└── latest/                 # 软链接/复制到最新
```

Immutable Rule IR-8: 输出不可覆盖
```
所有输出写入时间戳目录 output/run_YYYYMMDD_HHMMSS/
禁止覆盖历史版本
```

### 违反后果
- 结果丢失：无法追溯历史估值变化
- 调试困难：无法对比修正前后的差异
- 审计失败：无法提供历史运行证据

## 决策 9: 参数提取前置原则

### 背景
实践中发现建模阶段临时补充参数提取导致数据混乱。

### 选项分析

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 边建模边提取 | 灵活，按需获取 | 数据不完整，交叉验证缺失 |
| **B: 第一阶段完整提取** | 数据完整性有保障，可追溯 | 前期耗时较长 |

### 最终选择
**选项 B**: 所有计算参数必须在第一步（参数提取阶段）完成

工作流程：
```
PDF读取 → 参数提取与存档 → 参数确认 → 建模 → 模型审计 → 估值比对 → 差异记录
```

禁止行为:
```
在建模阶段发现数据缺失而临时补充参数提取
```

Immutable Rule IR-9: 参数提取前置
```
所有计算参数必须在第一步（参数提取阶段）完成并归档至 extracted_params_detailed.json
建模阶段禁止临时补充参数提取
```

### 违反后果
- 数据混乱：建模过程中发现的数据错误无法追溯
- 版本不一致：同一项目多次提取导致数据版本不一致
- 审计失败：无法证明建模使用的数据版本

## 决策 10: 综合体项目回退机制

### 背景
华润成都项目中，酒店部分数据不完整导致计算为0。

### 选项分析

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 严格计算，缺失为0 | 逻辑一致 | 综合体估值不完整 |
| **B: 招募估值回退** | 保证综合体估值完整性 | 需要标记回退来源 |

### 最终选择
**选项 B**: 当某业态计算为0但招募说明书有估值时，自动添加回退项目

```python
if hotel_val > 0 and asset_type_valuation.get("hotel", 0) == 0:
    all_projects.append(ProjectResult(
        name="酒店部分（招募说明书估值回退）",
        asset_type="hotel",
        valuation=hotel_val,
        noi_source="appraisal_fallback",  # 标记回退来源
        ...
    ))
```

### 违反后果
- 估值不完整：综合体估值缺失部分业态
- 无法对比：模型估值与招募总估值不可比
- 用户困惑：无法解释为何某业态估值为0
